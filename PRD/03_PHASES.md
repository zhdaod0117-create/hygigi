> Phase별로 나눠서 각각 "진짜 동작하는 제품"을 만듭니다.
> 전제: 기존 앱(투두/메모/로그인/Realtime)은 이미 배포 가능한 상태.

## Phase 1: MVP — 비동기 소통 코어 (주말 2~3번 분량)

### 목표
평일에 두 사람이 하루 3초~3분으로 감정·하루를 남기고, 서로의 기록에 반응할 수 있다.

### 기능
- [ ] 감정 날씨 체크인 (5종 탭 + 선택 한 줄, 실시간 반영, 강아지 말풍선 반응)
- [ ] 출퇴근/야근 상태 (4종 탭, 헤더에 서로 상태 표시)
- [ ] 오늘의 질문 교환일기 (질문 시드 90개, 작성/당일 수정, 둘 다 작성 시 공개 — RLS 강제)
- [ ] 읽음 표시 + 이모지 리액션 6종 (일기·감정로그 대상)
- [ ] 기록 타임라인 (날짜별 감정 로그 + 일기 아카이브)
- [ ] 하단 탭 네비게이션 (홈 / 일기 / 기록) + 홈 상단 "오늘의 우리" 스트립
- [ ] 강아지 반응 확장: 체크인/일기 완성/리액션 이벤트별 말풍선

### 데이터
- 신규: diary_questions, diary_entries, mood_logs, work_status_logs, reactions
- 기존 테이블 변경 없음. 신규 테이블 Realtime publication 추가

### 인증
- 기존 그대로 (매직링크 + 6자리 코드)

### "진짜 제품" 체크리스트
- [ ] 실제 DB 연결 (목업 데이터 X)
- [ ] 실제 인증 (하드코딩 X)
- [ ] RLS로 "둘 다 써야 공개" 서버 강제 확인 (다른 계정으로 직접 검증)
- [ ] Vercel 배포 + 두 사람 휴대폰으로 실사용 1주
- [ ] 기존 투두/메모 회귀 테스트 통과

### Phase 1 시작 프롬프트
```
이 PRD를 읽고 Phase 1을 구현해주세요.
@PRD/01_PRD.md
@PRD/02_DATA_MODEL.md
@PRD/04_PROJECT_SPEC.md

Phase 1 범위:
- 감정 날씨 체크인 + 출퇴근 상태 (탭 한 번, Realtime 반영)
- 오늘의 질문 교환일기 (둘 다 작성 시 공개 — 반드시 RLS로 강제)
- 리액션/읽음 표시, 기록 타임라인, 하단 탭 3개
- 강아지 말풍선 반응 확장

반드시 지켜야 할 것:
- 04_PROJECT_SPEC.md의 "절대 하지 마" 목록 준수
- 기존 테이블/기능 변경 금지, schema.sql에 추가만
- 실제 DB 연결 (목업 X), 서버 액션은 { data, error } 반환 패턴 유지
```

## Phase 2: 확장 — 주말과 일상 (Phase 1 실사용 2주 후)

### 전제 조건
- Phase 1이 배포되어 두 사람이 실제로 1~2주 사용, 계속 쓰게 되는지 확인

### 목표
평일에 쌓은 것이 주말 데이트로 이어지고, 감정 기록을 달 단위로 돌아볼 수 있다.

### 기능
- [ ] 주말 계획 보드 (평일에 "이번 주말에 하고 싶은 것" 적립 → 주말에 체크)
- [ ] 점심/저녁 메뉴 공유 (사진 1장 + 한 줄, Supabase Storage)
- [ ] 감정 캘린더 (월 뷰, 두 사람 날씨 나란히)

### 추가 데이터
- weekend_wishes (또는 todos 확장 — P2 시작 전 결정, 02_DATA_MODEL 미결 항목)
- meal_posts + Storage 버킷(이미지, 워크스페이스별 폴더 + RLS)

### 통합 테스트
- Phase 1 기능(일기 공개 규칙, Realtime) 회귀 확인

## Phase 3: 고도화 — 알림과 회고

### 전제 조건
- Phase 1+2 안정 운영, "확인하러 들어오는" 습관이 생겼는지 관찰

### 목표
앱을 열지 않아도 상대의 신호를 놓치지 않고, 쌓인 기록이 선물이 된다.

### 기능
- [ ] 웹푸시 알림 (상대 체크인/일기 작성/리액션 — iOS PWA는 홈 화면 추가 필수라는 안내 포함)
- [ ] 월간/연간 리캡 ("7월의 우리: ☀️ 12일, 야근 8번, 일기 21편")
- [ ] 커스텀 질문 (질문 뱅크에 우리만의 질문 추가)
- [ ] 기념일/디데이

### 주의사항
- 웹푸시: iOS 16.4+ 홈 화면 PWA에서만 동작. VAPID 키 관리 필요.
- 리캡 집계 쿼리는 로그 테이블 설계 덕에 추가 마이그레이션 불필요.
